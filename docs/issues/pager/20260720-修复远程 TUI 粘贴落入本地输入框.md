# Issue: 修复远程 TUI 打开时粘贴落入 grok-pi 输入框

## 用户症状

- 远程 TUI（extension host 渲染的 Pi 交互组件，如带输入框的 LoginDialog）打开时，粘贴（Cmd+V / 终端 bracketed paste）不会进入远程组件的输入框。
- 粘贴的文本落入 grok-pi 自己的 prompt 输入框。

## 根因

`AppView::handle_input_at_with_paste_provenance` 中的 Remote TUI 拦截只处理 `Event::Key`（let-chain 里 `let Some(key) = key_event`）。crossterm 的 `Event::Paste`（终端 bracketed paste，或 event_loop 快速按键合并产生的合成 paste）不满足该条件，直接落到后续本地 prompt 编辑路径，文本插入 grok-pi 输入框。

## 设计

| 关注点 | 改动 |
|---|---|
| 粘贴转发 | Remote TUI 激活时拦截 `Event::Paste(text)`，包裹 Pi-TUI bracketed paste 标记 `\x1b[200~ … \x1b[201~` 后经 `Effect::RemoteTuiInput` 转发。该标记与 Pi 自身约定一致（interactive-mode `pasteToEditor` 即用此格式），Pi 的 input/editor 组件会将多行内容作为粘贴整体插入，换行不会触发 submit。 |
| 按键路径 | 原有 Esc→cancel、`remote_tui_key_sequence` 转发逻辑不变，仅移入 `if let Some(key)` 分支。 |
| 传输 | 数据经 ACP `pi/ui/remote_tui/input` → adapter 追加 keyfile JSONL（`\u001b`/换行均被 JSON 转义）→ extension `drainKeys` → 组件 `handleInput`。 |

## 边界

- 空粘贴不转发，落回原路径（本地亦无副作用）。
- 列表型组件（如 probe list）收到粘贴序列为无害 no-op；此前则会污染本地 prompt。
- 远程会话关闭后粘贴恢复为本地 prompt 行为。
- 不改 Pi 源码、不改 RPC；adapter 保持 headless。

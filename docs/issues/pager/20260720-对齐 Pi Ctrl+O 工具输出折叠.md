# Issue: 对齐 Pi Ctrl+O 工具输出折叠

## Goal

让 `grok-pi` 的完整 Pager 模式下 `Ctrl+O` 具备可配置的展开范围：默认 `write_edit`，可选全部工具输出。

## 背景/问题

Pi 原生将 `Ctrl+O` 绑定为 `app.tools.expand`，统一更新所有 tool-output 组件；Pager 原本将该按键用于 Grok 的 always-approve。Pi external profile 必须保持 Pi 的快捷键语义，且不能影响消息、thinking、审批或 minimal 模式的既有路由。

## 验收标准 (Acceptance Criteria)

- [x] WHEN `grok-pi` 处于非 minimal 的 agent 页面且按下 `Ctrl+O`，系统 SHALL 按 F2 设置切换目标 ToolCall 的展开状态。
- [x] WHERE 默认 `write_edit`，系统 SHALL 折叠/展开 `write` 与 `edit` 输出。
- [x] WHERE 用户选择 `all_tools`，系统 SHALL 折叠/展开全部工具输出。
- [x] IF 消息或 thinking block 存在，THEN Ctrl+O SHALL 不改变其显示模式。
- [x] IF 非 Pi profile 或 minimal 模式，THEN 系统 SHALL 保留既有快捷键路由。

## 实施阶段

### Phase 1: 规划和准备
- [x] 分析 Pi keybindings 与 `InteractiveMode.setToolsExpanded()`。
- [x] 追踪 Pager 的 `Ctrl+O`、外部 profile 与 scrollback 折叠实现。
- [x] 设计单次扫描/单次 generation 更新方案。
- [x] 确认默认范围为 `write_edit`，而非 Read。

### Phase 2: 执行
- [x] 在 ScrollbackState 添加工具调用范围的批量切换。
- [x] 在 Pi external、非 minimal 路由中拦截 Ctrl+O。
- [x] 添加 profile 路由回归测试。
- [x] 添加 F2 Enum 设置、持久化与 rollback 路径。

### Phase 3: 验证
- [ ] 聚焦单元测试（Pager 测试目标仍被既有跨模块测试编译问题阻断）

## 关键决策

| 决策 | 理由 |
|------|------|
| 默认匹配 `RenderBlock::ToolCall::Edit` | Pi 的 `write` 与 `edit` 都投影为 Edit；Read 不会被展开。 |
| `all_tools` 作为 F2 选项 | 保留 Pi 原生全部工具输出的可选行为。 |
| 扫描后批量失效缓存 | 只做一次 generation 与 group 状态更新，避免逐条重建布局。 |
| 只在 external 且非 minimal 拦截 | 保留普通 Grok Ctrl+O、minimal transcript/interject 语义。 |

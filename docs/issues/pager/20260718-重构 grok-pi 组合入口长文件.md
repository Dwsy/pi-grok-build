# Issue: 重构 grok-pi 组合入口长文件

## Goal

将 `grok-pi` 组合入口按 CLI、Pi 会话路径和 session-tree bridge 三个纯职责拆分为私有模块，同时保持入口只负责 Pi RPC、ACP 与生产 Pager 的组合。

## 背景/问题

盘点确认 `crates/codegen/xai-grok-pager-bin/src/bin/grok-pi.rs` 有 625 行。它同时容纳 CLI 解析及 Pi 参数投影、会话目录解析、临时 Pi extension 注入和进程组合，降低了组合入口的可读性。

范围只限该二进制。相邻长文件不在本 Issue 中：同 crate `src/main.rs`（2,949 行）属于标准 Grok 入口；`pi-grok-adapter/src/pi_adapter.rs`（3,538 行）与 `model.rs`（1,479 行）属于 headless 协议层；Pager 的 `app/mod.rs`、`acp/mod.rs` 与 `mermaid_worker.rs` 是已声明的原生接缝。

## 验收标准 (Acceptance Criteria)

- [x] `grok-pi.rs` 保留唯一 composition root：`PiRpc::spawn` → `PiBootstrap::load` → `PiAgent`/ACP channel → `AcpConnection::external` → `run_external`。
- [x] CLI 解析、复合 Pi 短参数归一化与 Pi startup flags 投影移至私有 `grok_pi/cli.rs`，并保留既有参数顺序与 `--` 后参数不重写的行为。
- [x] Pi session 目录解析移至私有 `grok_pi/session_paths.rs`，并保留最后一个 `--session-dir` 优先、环境变量回退、相对路径相对 Pi cwd 的语义。
- [x] 临时 `ctx.navigateTree`/`ctx.setLabel` extension 写入移至私有 `grok_pi/tree_bridge.rs`，并保留 `.ts` 后缀、`NamedTempFile` 生命周期和官方 extension API。
- [x] 不修改 `pi-grok-adapter`、Pager 或包内 Pi 源码；不创建 terminal、widget、输入循环或第二套 UI。
- [x] `cargo test -p xai-grok-pager-bin --bin grok-pi` 与 `cargo check -p xai-grok-pager-bin --bin grok-pi` 成功；差异仅包含本 Issue 必需的文件。

## 关键决策

| 决策 | 理由 |
|------|------|
| 仅拆 `grok-pi.rs` | 625 行文件是当前组合入口的直接可控目标；混拆 adapter/Pager 会扩大协议和 UI 回归风险。 |
| 根文件保留组合逻辑 | 进程、ACP channel 和 Pager 生命周期共同构成唯一正确的所有权拼接点。 |
| 私有模块留在 `src/bin/grok_pi/` | 保持二进制私有实现细节，不扩张 crate 公共 API。 |
| 临时 tree bridge 不下沉 adapter | 注入与保活是二进制生命周期策略；adapter 仍只翻译 JSONL RPC ↔ ACP。 |

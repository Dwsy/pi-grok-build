# Issue: 适配 Pi Config 资源管理

## Goal

在 `grok-pi` 的原生 Grok Pager F2 设置中提供 Pi `config` 等价的资源管理入口：准确呈现 Pi 的全局/项目作用域和资源启用状态，并以 Pi 的配置格式持久化变更。

## 背景/问题

当前 `grok-pi` 已将 Pi 运行期能力映射到原生 Pager，但资源启用状态只能在独立的 Pi TUI 中通过 `pi config` 管理。用户需要在同一 Grok 原生表面中检查和切换扩展、技能、提示模板与主题等已发现资源。此工作必须保持 Pager 为唯一 TUI、adapter 无 UI、Pi 配置语义与项目可信边界为唯一真相。

## 验收标准 (Acceptance Criteria)

- [x] WHEN 用户在 F2 中选择 Pi Config，系统 SHALL 打开原生 Pager 的资源配置界面，而不创建第二套 TUI。
- [x] WHERE 用户在全局或已信任项目作用域中查看资源，系统 SHALL 与 Pi `config` 的发现结果和启用状态一致。
- [x] WHEN 用户切换某个可配置资源，系统 SHALL 用 Pi 相同的设置格式持久化该作用域，并清楚提示是否需要重启/重新加载。
- [x] IF 项目未获 Pi 信任，系统 SHALL 不读取、加载或修改项目本地资源。
- [x] WHEN 变更完成，系统 SHALL 不扩展 adapter 的终端/UI 职责，也不修改 Pi RPC 协议。

## 关键决策

| 决策 | 理由 |
|------|------|
| Rust compatibility layer 是 Pi config 的本地实现 | 用户明确授权以 Pi 源码为行为参考；不得改 `pi-main`，也不得在 adapter 实现 UI。 |
| F2 只作为入口，资源面板复用 Pager 原生组件 | 满足单一 TUI 与 native-surface 架构约束。 |
| 首切片实现资源清单与启停，不将 `pi install/remove/update` 混入 | 安装源管理是独立的 package lifecycle，避免引入网络与受管目录删除风险。 |
| 项目资源受 Pi trust gate 约束 | Rust compatibility layer 必须读取 Pi trust store 并在未信任时拒绝项目扫描与写入。 |

## Notes

### 研究结论

Pi `config` 只管理 `extensions`、`skills`、`prompts`、`themes` 四类已解析资源；它支持 user/global 与 trusted project 作用域。全局是二态启用，项目层是 `inherit` / `load` / `unload` 三态。资源可能来自自动发现目录、settings 显式路径或 npm/git/local package；Pi 使用 `SettingsManager` 的迁移、锁、字段级合并与 trust guard 写入 settings。

`pi config` 自己创建 `ProcessTerminal` 与 Pi TUI，故不能从 F2 启动或嵌入。现有 RPC 只能给出已加载 command catalog，不能表示 disabled item、theme、资源来源、package filter、scope、trust 或配置 mutation。Pager F2 可以安全地增加一个 native entry/action，但无法在不复制 Pi 逻辑的情况下取得或变更真实配置。

用户已授权改为 Rust compatibility layer：以 bundled Pi 的配置解析、scope override 和 trust 语义作为行为参考，在 Pager 原生面板中实现，不修改 `pi-main`，不在 adapter 中实现 UI。实现已提供 F2 → Pi resources 原生 modal：全局二态与 trusted project 的 inherit/load/unload 三态、自动目录/显式路径/已安装 npm/git/local package 发现、Pi `trust.json` gate、Pi 风格 `+/-` settings mutation 与 restart/`/reload` 提示。它刻意不包含 `pi install/remove/update`，后者仍是独立 package lifecycle。

### 手工原生 TUI 验证

2026-07-18 使用刚构建的 `target/debug/grok-pi --no-session -- --offline` 在隔离 tmux session 中验证：进入会话后按 F2，搜索 `Pi resources` 可匹配唯一的 Agent & Approval 设置行；两次 Enter 打开 Pager 原生 `Pi resources` modal；资源清单显示 global 的 settings/auto 来源、resource type 和 restart/reload 提示；按 Tab 成功切换至 Project 的 inherit/load/unload 作用域。验证期间未切换资源，未改动用户 Pi settings。

# Issue: 适配第三方扩展 RPC 检测

## Goal

在不修改 Pi 源码或第三方扩展文件的前提下，让 grok-pi 为可由 Remote TUI 承载的扩展提供 TUI 兼容外观，同时持续使用 Pi 的真实 JSONL RPC 协议。

## 背景/问题

部分第三方扩展以 `process.argv.includes("--mode") && process.argv.includes("rpc")` 将 RPC 等同于无 UI，并直接关闭其交互能力。Pi 仍必须以 RPC 模式运行，且其官方扩展上下文还会提供 `ctx.mode === "rpc"`。

## 验收标准 (Acceptance Criteria)

- [x] npm/Node Pi host 可通过官方 `dist/rpc-entry.js` 启动；Pi 内部仍运行 RPC，第三方扩展 argv 不含外层 `--mode rpc`。
- [x] Remote TUI 启用时，兼容扩展在 Pi 的 `ExtensionRunner.setUIContext()` 边界将扩展可见的 `ctx.mode` 投影为 `"tui"`，不修改 Pi 源码或全局 `process`。
- [x] Remote TUI 关闭或无法解析 `rpc-entry.js` 时，保留原始 RPC 启动/`ctx.mode` 语义并记录安全回退。
- [x] adapter 窄测试和 grok-pi binary check 通过；diff 仅包含启动与注入接缝、测试和本文档。

## 关键决策

| 决策 | 理由 |
|------|------|
| 使用 Pi 官方 `rpc-entry.js` | 该入口把 `--mode rpc` 仅传入 `main()`，不暴露于扩展的 OS argv。 |
| 只在 Remote TUI 已启用时投影 `ctx.mode` | `tui` 外观必须有可承载 `custom()` 的 host；否则保持真实 RPC 语义。 |
| patch `ExtensionRunner.setUIContext` | 在 Pi 的正式 UI 绑定边界集中处理，不篡改 `process`、不重写第三方扩展源码。 |
| 对无法识别的 Pi host 回退 | `--pi-bin` 允许 Bun/自定义启动器；不能猜测其 npm 包布局。 |

## Notes

Pi 的正式 `ExtensionContext.mode` 会在 RPC 模式下绑定为 `"rpc"`；此兼容层只为已启用 Remote TUI 的 grok-pi 子进程呈现 `"tui"`。Pi 核心仍保持 `appMode === "rpc"`，stdin/stdout JSONL transport 不变。已用已安装 Pi 0.80.10 验证 `rpc-entry.js --version`，并通过 jiti 实例化兼容扩展，确认 `ExtensionRunner.setUIContext(..., "rpc")` 投影为 `tui`。

# Issue: 修复 Pi RPC 通知与 Widget 显示隔离

## Goal

将 Pi RPC 的 `notify`、`setStatus` 与 `setWidget` 映射到不互相覆盖、且保留 widget placement 语义的原生 Grok Pager 表面。

## 背景/问题

Pi RPC 已传输 `widgetKey`、`widgetLines` 和 `widgetPlacement`。Pager 却在
`AppView::refresh_external_ui_surface()` 中把 widgets 和 statuses 合并为一条
`sticky_toast`；`AgentView::draw()` 再将该槽固定绘制在 scrollback 右下角。
同时 `notify` 使用单槽 transient toast，后到通知会覆盖前一条。

## 验收标准 (Acceptance Criteria)

- [ ] 多条 `notify` 按到达顺序显示，不会彼此替换或覆盖。
- [ ] 同时存在的 `setStatus` 与多个 `setWidget` 不混合为单条右下角文本。
- [ ] `setWidget(..., { placement: "aboveEditor" })` 与 `belowEditor` 分别显示在对应原生编辑器区域。
- [ ] 清除指定 status/widget key 只移除目标内容。
- [ ] 适配器维持 headless，不新增 Ratatui/Crossterm 依赖或自绘终端。

## 实施阶段

### Phase 1: 规划和准备
- [x] 分析 Pi RPC UI 协议与 Pager 的映射、状态和渲染路径。
- [x] 确认根因：所有持久外部 UI 被压缩进单一 `sticky_toast`，通知使用单槽 toast。
- [x] 确定复用的 Pager 原生布局区域与最小数据流。

### Phase 2: 执行

## 关键决策

| 决策 | 理由 |
|------|------|
| 保留 Pi RPC 的 key 和 placement | 协议已提供足够语义，不能在 Pager 侧丢弃。 |
| 复用 Pager 原生表面 | 符合“不创建第二套 TUI”的架构约束。 |
| 通知采用队列 | `notify` 是瞬态事件，不能使用会替换内容的单槽状态。 |

# Issue: 重构为Grok根仓库

## Goal

使 `grok-build-main/` 成为项目唯一 Git 根目录和 GitHub fork 工作树；Pi 源码、交付脚本、文档与验证资产全部收拢在该根目录。

## 背景/问题

原交付采用一个外层包装仓库承载 `grok-build-main/` 与 `pi-main/`，导致 Grok 源码没有独立 Git 历史，且 GitHub 仓库不显示为上游 Grok Build fork。

## 验收标准 (Acceptance Criteria)

- [x] WHEN 用户进入仓库，系统 SHALL 在项目根直接使用 Grok Build 的 Git 历史和构建入口。
- [x] WHERE Pi、脚本、文档与验证资产存在，系统 SHALL 将其放在同一 Grok 根目录下。
- [x] IF GitHub 仓库发布，THEN 它 SHALL 显示为 `xai-org/grok-build` 的 fork 并以当前交付覆盖其 main 分支。

## 实施阶段

### Phase 1: 规划和准备
- [x] 分析需求和依赖：现有 GitHub 仓库不是 fork，Git push 无法改变 fork 归属。
- [x] 设计技术方案：重命名旧普通仓库、创建上游 fork、建立独立 Git 根并迁移交付资产。
- [x] 确定实施计划。

### Phase 2: 执行
- [x] 建立 `grok-build-main/.git`，基线为 `c68e39f60462f28d9be5e683d9cbe2c57b1a5027`。
- [x] 将 Pi、脚本、文档、CI 和验证日志迁移到 Grok 根；更新脚本与 README 路径。
- [x] 创建根仓库提交并 force-push 到 GitHub fork。

### Phase 3: 验证
- [ ] 单元测试
- [ ] 集成测试
- [ ] 代码审查

### Phase 4: 交付
- [ ] 更新文档
- [ ] 创建 PR
- [ ] 合并主分支

## 关键决策

| 决策 | 理由 |
|------|------|
| GitHub fork 独立于 Git push | GitHub 的 fork graph 只能通过 GitHub 创建 fork，不能由 force push 改写。 |
| 保留上游 remote | `upstream` 继续指向 `xai-org/grok-build`；`origin` 指向 `Dwsy/pi-grok-build` fork。 |

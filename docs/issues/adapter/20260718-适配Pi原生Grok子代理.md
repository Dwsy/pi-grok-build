# Issue: 以 Pi extension 驱动原生 Grok 子代理 TUI

## Goal

新增一个由 Pi extension 独占子代理生命周期的实现，使 Grok Pager 复用既有的原生 Subagent scrollback、Tasks Pane、取消操作与 child AgentView，而不修改 Pi Core、不新增第二套 TUI，且不让 `pi-grok-adapter` 管理 agent runtime。

## 背景/问题

原版 Grok Pager 已消费 `x.ai/session/update` 下的 `SubagentSpawned`、`SubagentProgress`、`SubagentFinished`，并据此维护 child view 与 Tasks Pane。当前 Pi RPC 只投影普通工具卡；`@tintinweb/pi-subagents` 的事件粒度和 Pi-TUI UI 依赖无法提供原生 Grok 所需的 child transcript/replay 协议。

本任务不 fork Grok UI，也不把 adapter 变成 scheduler。新 extension 负责创建、运行、取消与追踪独立 Pi child session；adapter 只把经过验证的 bridge 事件转换到 Pager 已有的协议，并将 Pager 的取消请求路由回 extension。

## 验收标准 (Acceptance Criteria)

- [ ] WHEN 父 Pi agent 启动子代理，系统 SHALL 在原生 Grok scrollback 创建可交互的 SubagentBlock，并在 Tasks Pane 的 `Subagents` 分组显示同一任务。
- [ ] WHEN 子代理运行，系统 SHALL 用 Pi 可证明的 duration、turn/tool、usage 与活动数据更新原生子代理状态；未知值不得伪造成精确统计。
- [ ] WHEN 子代理完成、失败或取消，系统 SHALL 发送一次终态并清除 Pager 的运行状态；重复或乱序事件不得产生重复 block 或悬挂任务。
- [ ] WHEN 用户在原生 Tasks Pane 对运行中子代理按 `x`，系统 SHALL 将 `x.ai/subagent/cancel` 路由给 Pi extension，并仅在 extension 确认终止后呈现 `cancelled`。
- [ ] WHEN 用户从 scrollback 或 Tasks Pane 打开 child，系统 SHALL 呈现真实 Pi child transcript；恢复后可按同一 contract 回放，而非空 AgentView。
- [ ] WHERE `pi-grok-adapter` 参与桥接，系统 SHALL 保持 headless、无终端/键盘/调度职责；Pager SHALL 继续是唯一可见 TUI。
- [ ] IF bridge 版本、session 关联或 child stream 无法验证，THEN 系统 SHALL 拒绝/降级该事件并记录诊断，而非伪造 Grok 子会话状态。

## 关键决策

| 决策 | 理由 |
|------|------|
| 新写 Pi extension，不以 `@tintinweb/pi-subagents` 为协议基础 | 后者是进程内 Pi extension，但没有 Grok 所需的 child-session stream/replay contract，且原生 UI 依赖 Pi TUI。 |
| Pager 原生组件零重写 | `SubagentBlock`、Tasks Pane、child AgentView 和 cancel 已存在；复用它们符合唯一 TUI 边界。 |
| Adapter 只做协议投影 | Pi extension 必须拥有 child session、取消和状态；adapter 持有这些会造成 agent core 分叉。 |
| 以版本化 custom message bridge 起步，先用 fixture 证明 | 不修改 Pi RPC；只有确认 custom message 的实时线形和 child ACP route 后才能承诺实现细节。 |
| child transcript 是全兼容的硬门槛 | 仅 lifecycle 事件会产生空 child AgentView，只能算部分适配。 |
| extension 仅在 `PI_GROK_SUBAGENTS=1` 时注册 | `grok-pi` 通过 `SpawnConfig.env` 向其 Pi RPC child 单独注入该变量；手动 Pi、全局 Pi 与其他宿主即使显式加载源文件也不会注册工具或命令。 |

## Notes

`SubagentProgress` 采用约两秒节流；桥接不得按 token 输出频率刷 Pager。原版 Pager 用 child session ID 路由 child updates，因此 parent lifecycle 与 child stream 必须共享可验证的 identity 映射。`x.ai/subagent/cancel` 的响应不能直接声称成功；UI 终态必须由 extension 产出的最终 finish 事件确认。

Bridge fixture 已通过全局 Pi `0.80.10` 验证：在 `--mode rpc --no-session --no-extensions --extension <probe.ts>` 下，`pi.sendMessage({ customType: "pi-grok-subagent/v1", content, display:false, details })` 先后输出 `message_start` 与 `message_end`，两者均完整保留 `role:"custom"`、`customType`、`display:false` 与结构化 `details`。因此 adapter 可在既有 Pi RPC event stream 中识别这个版本化 envelope，而不需扩展 Pi RPC。

Child ACP 路由的源码事实：adapter 当前 `send_update()` 只以 root Pi session ID 构造 `acp::SessionNotification`（`pi_adapter.rs:375-386`）；Pager 的 `AcpClientMessage::SessionNotification` 按 `request.session_id` 解析，若命中已创建 child view，则直接将普通 `SessionUpdate` 交给 child 的 `AgentSession` 与 scrollback（`xai-grok-pager/src/app/acp_handler/mod.rs:70-100, 460-540`）。因此可新增显式 session-ID 的薄发送方法；它不要求 adapter 了解 child runtime。尚需编译级 fixture 与 replay API 设计。

## Bridge contract v1（提案，实施前先以 fixture 锁定）

**传输：**extension 对每个事件调用 `pi.sendMessage({ customType: "pi-grok-subagent/v1", content: "", display: false, details: envelope }, { triggerTurn: false })`。adapter 仅在 `message_end` 应用，忽略同一 payload 的 `message_start`；空 `content` 防止 child 流污染 parent LLM context，`details` 是唯一协议载荷。

**公共 envelope：**

```ts
{
  version: 1,
  sequence: number,          // parent session 内严格单调
  replay: boolean,           // resume 重放时为 true
  kind: "spawned" | "progress" | "child_update" | "finished",
  parentSessionId: string,   // 必须匹配 adapter 当前 Pi root session
  subagentId: string,
  childSessionId: string,    // Pi child AgentSession.sessionId，Pager 路由键
  payload: object,
}
```

| kind | payload 必填字段 | adapter 行为 |
|------|------------------|--------------|
| `spawned` | `parentToolCallId`、`description`、`subagentType`、`background`、`prompt`；可选 `model`、`persona`、`role`、`capabilityMode` | 先用 `parentToolCallId` 发 parent `ToolCallUpdate.raw_input={variant:"Task",task_id:subagentId,run_in_background:background}`，使 Pager 正确判定前台/后台；再发 parent `x.ai/session/update:subagent_spawned`；最后发 child `UserMessageChunk(prompt)`。 |
| `progress` | `durationMs`、`turnCount`、`toolCallCount`、`toolsUsed`、`errorCount`；可选 `tokensUsed`、`contextWindowTokens`、`contextUsagePct` | 发 parent `x.ai/session/update:subagent_progress`，最多每两秒一次；未知 usage 不伪造。 |
| `child_update` | `update`，其 discriminant 为 `user`、`assistant_delta`、`thinking_delta`、`tool_call`、`tool_update`、`tool_result` 或 child session event | 转换为**指定 `childSessionId`**的标准 ACP `SessionNotification`，使既有 child `AgentView` 消费。 |
| `finished` | `status` (`completed`/`failed`/`cancelled`)、`durationMs`、`turns`、`toolCalls`、可选 `error`、`tokensUsed`、`output` | 只接受一次终态；发 parent `x.ai/session/update:subagent_finished`，以 Pager 原生终态收束 UI。 |

**持久化与 replay：**child `SessionManager` 必须启用持久化；extension 在 parent Pi session 写入小型 agent index（id、child session file、状态、生命周期 sequence），而非把全部 child transcript 复制到 parent context。parent `/resume` 后 extension 读取 index、重建 lifecycle，并从 child session 文件发出 `replay:true` 的 user/assistant/thinking/tool stream。第一版可在 root replay 时预热所有 child transcript；若要和原版的“打开才懒加载”完全等价，需新增一个声明化的 External-profile `x.ai/subagent/load` seam，不能靠 Pager 猜测本地 `.grok` 文件。

**取消：**Pager 已发 `x.ai/subagent/cancel`。adapter 只把 `{subagentId}` 转为隐藏 extension command `__pi_grok_subagent_cancel` 的 Pi RPC `prompt`；extension 调用它所拥有的 child `AgentSession.abort()`，随后由 `finished{status:"cancelled"}` 确认 UI 终态。adapter 不直接取消 child，也不在取消 RPC 成功时伪造完成。

# Issue: 隔离 grok-pi 产品版本

## Goal

让 `grok-pi` 的版本、更新比较和发布构建只使用项目自己的 `GROK_PI_VERSION`，不再读取或受上游 Grok `GROK_VERSION` / workspace crate version 影响。

## 背景/问题

`grok-pi` 与上游 Grok 共用 `xai-grok-pager-bin` 和 `xai-grok-update`。此前 release workflow 以 `GROK_VERSION` 注入产品版本，且 Pi 更新路径从 `xai_grok_version::installed()` 读取当前版本；上游同步或 Grok release stamp 可覆盖/污染 `grok-pi` 的版本判断。

已确认当前发布版本为 `0.0.4`，本地 `grok-pi --version` 来自 `GROK_PI_VERSION` / git tag。修复目标是让所有 Pi 产品路径显式传递该值，保持上游 Grok 版本语义独立。

## 验收标准 (Acceptance Criteria)

- [x] `grok-pi --version` 的构建脚本只接受 `GROK_PI_VERSION`，不回退到 `GROK_VERSION` 或 `CARGO_PKG_VERSION`。
- [x] GitHub release workflow 仅注入 `GROK_PI_VERSION`。
- [x] `grok-pi update`、Welcome 自动检查和 Ctrl+U 安装均使用组合入口传入的 `GROK_PI_VERSION`，不读取 `xai_grok_version::installed()`。
- [ ] `cargo test -p xai-grok-pager-bin --bin grok-pi` 通过（当前被既有 adapter API 漂移阻断，见 Notes）。
- [x] `cargo check -p xai-grok-pager-bin --bin grok-pi` 通过。
- [x] 审查 diff，确认没有改变 adapter 的 headless 边界或创建额外 TUI。

## 关键决策

| 决策 | 理由 |
|------|------|
| 不修改 `xai-grok-version` | 该 crate 是上游 Grok 的通用版本来源；改变它会影响 workspace 的 trust、遥测和普通 Grok 更新语义。 |
| 通过 `ExternalRunConfig` 显式传递版本 | `run_external` 已是生产 Pager 与 Pi composition 的窄边界；不会让 Pager 推断或读取 Pi 版本。 |
| release workflow 只设置 `GROK_PI_VERSION` | 防止 Pi 发布 stamp 改变上游 `GROK_VERSION` 的 release-only 行为。 |

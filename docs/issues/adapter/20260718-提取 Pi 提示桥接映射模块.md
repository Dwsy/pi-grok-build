# Issue: 提取 Pi 提示桥接映射模块

## Goal

将 ACP prompt 与 Pi prompt/queue/built-in Bash 之间的无状态映射提取为私有模块，保持 active-turn 默认 follow-up、send-now steer、client prompt ID 回传与 Bash 文本显示语义不变。

## 背景/问题

在工具与上下文投影提取后，`pi_adapter.rs` 仍有 2,286 行。`1867–1988` 行是一个独立的 prompt bridge：识别 Pager 传来的 `bash_command` 元数据、将 ACP content blocks 映射为 Pi message/images、选择 `steer` 或 `followUp`、把行为映射为 queue lane、在 ACP PromptResponse 上回写 `promptId`、以及格式化 Pi 的直接 Bash 结果。

这些都是无状态协议映射；`PiAgent` 本身继续拥有 active prompt、RPC、queue mirror reserve、ToolCall 生命周期和 ACP send_update，因此提取不改变异步所有权。

不在本轮范围：`PiAgent` 事件循环、`acp::Agent` trait、session/tree bridge、extension QuestionView 映射、model/command catalog 及 `model.rs`。

## 验收标准 (Acceptance Criteria)

- [x] `prompt_bridge.rs` 只依赖 `model::json_text`、`queue_bridge::QueueLane`、ACP 与 serde JSON；不依赖 Pi RPC、PiAgent 状态、terminal 或 Pager UI。
- [x] `pi_adapter.rs` 保留 prompt 的 RPC、active prompt/queue mirror 变更、ToolCall 发送和 error handling；只调用私有映射函数。
- [x] ACP text/image/resource prompt 转 Pi payload、`bash_command` 识别、promptId echo、follow-up/steer 规则、QueueLane 和 Bash 文本格式完全不变。
- [x] 相关 4 项单测与实现同置，`cargo test -p pi-grok-adapter`、adapter/binary `cargo check` 与 `./build.sh` 成功。
- [x] 不修改 Pager、包内 Pi、`queue_bridge` 的队列语义或 adapter 公共 API。

## 关键决策

| 决策 | 理由 |
|------|------|
| 只提取无状态 prompt mapping | prompt 生命周期、queue mirror 与 RPC 所有权仍集中在 PiAgent，避免时序回归。 |
| `QueueLane` 直接复用 | 它是既有 queue bridge 的 crate-private 领域类型，不创建同义枚举。 |
| direct Bash 格式与 prompt bridge 同置 | `!command` 由 ACP prompt 进入 Pi Bash；文本结果是该边界的直接输出映射。 |
| 测试与实现同置 | 锁定 promptId、follow-up/steer 和 bash meta 协议。 |

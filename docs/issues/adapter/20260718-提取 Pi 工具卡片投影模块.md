# Issue: 提取 Pi 工具卡片投影模块

## Goal

将 `pi_adapter.rs` 中纯 Pi 工具输入/输出到 ACP 原生工具卡片的投影提取为私有模块，保持 JSONL RPC ↔ ACP payload 与 live/history 事件时序不变。

## 背景/问题

长文件盘点显示 `crates/codegen/pi-grok-adapter/src/pi_adapter.rs` 有 3,539 行。其 `2184–2741` 行是一组独立的工具卡片投影：内容块、Edit diff、工具分类、read/bash/grep/find/ls 的 `raw_input`/`raw_output` 归一化及 grep 解析。

该职责不拥有 RPC、`PiAgent` 状态、ACP channel 或终端表面，适合先做行为保持的提取。已有 `queue_bridge.rs` 和 `todo_bridge.rs` 是相同 adapter 内私有纯投影模块的先例。

不在本轮范围：`PiAgent` 生命周期与 `acp::Agent` trait、session/tree bridge、queue/todo bridge、context-info 投影，以及 `model.rs` 的 tree/catalog/RPC 解码。它们具有更大的异步或会话语义回归面。

## 验收标准 (Acceptance Criteria)

- [x] `tool_projection.rs` 仅依赖 `model::{PiToolContent, json_text, string}`、ACP、serde JSON 与 `IndexMap`；不引入 terminal、widget、renderer、keyboard 或 adapter 公共 API。
- [x] `pi_adapter.rs` 继续拥有 `tool_args` 生命周期、Pi 事件顺序、Todo Plan 投影与 ACP `send_update`；只调用私有工具投影函数。
- [x] history replay 与 live `tool_start`/`tool_update`/`tool_end` 对 `read`、`bash`、`edit`、`write`、`grep`、`find`、`ls` 保持相同的 ACP kind、content、`raw_input`、`raw_output` 形状。
- [x] 原有工具投影测试随实现迁移，`cargo test -p pi-grok-adapter`、`cargo check -p pi-grok-adapter` 与 `cargo check -p xai-grok-pager-bin --bin grok-pi` 成功。
- [x] 差异只包含 adapter 私有模块、其模块声明、`pi_adapter.rs`、本 Issue，以及先前未提交的 grok-pi 重构文件；不修改 Pi/Pager 源码。

## 关键决策

| 决策 | 理由 |
|------|------|
| 只提取工具投影 | 这组函数是纯 payload 转换；不会拆散共享的 `PiAgent` 异步生命周期。 |
| 模块保持 crate 私有 | 生产外部 API 仍仅为 `PiAgent`、`PiBootstrap`、`PiRpc` 与既有 session 重导出。 |
| 测试与实现同文件 | 既有 `queue_bridge.rs`/`todo_bridge.rs` 模式；测试直接锁定 ACP JSON 兼容形状。 |
| 不同时拆 `model.rs` | 避免同时变更 session JSONL 扫描、tree 展平与工具协议投影。 |

# Issue: 修复多区域 Edit 行号投影

## Goal

让 Pi `edit` 的多个不相邻 `edits[]` 区块在 Grok Pager 原生 Edit 卡片中显示实际文件行号，而非每个区块都从第 1 行开始。

## 背景/问题

截图复现：一个 Pi `edit` 调用携带多个 `edits[]` 时，Pager 每个 diff hunk 的 gutter 显示 `1…`。

当前数据流为：`Pi edit` 在结果 `details.patch` 中生成带真实 hunk 坐标的 unified patch；adapter 的 `edit_diff_content` 只从 tool args 生成多个 ACP `Diff(oldText, newText)`，没有 `Diff.meta` 行号；Pager `extract_edit_hunks` 在这类 fallback payload 上按协议默认 `start_line = 1`。Pager renderer 已验证会尊重 `Diff.meta.new_line`，因此无需修改终端渲染器或 Pi RPC。

不在范围：改 Pi edit tool/RPC、改 Pager diff renderer、用 adapter 读取或重写用户文件。

## 验收标准 (Acceptance Criteria)

- [x] Pi 返回 `details.patch` 且一个 edit 有多个不相邻替换时，adapter SHALL 从该 patch 投影每个 replacement 的绝对行号到 ACP Diff meta。
- [x] Pager 接到这些 Diff 后，hunk gutter SHALL 显示对应的绝对行号，而不是每个 hunk 重置为 1。
- [x] 无 patch、不可解析 patch 或无法唯一定位 replacement 时，adapter SHALL 保留原有无 meta fallback，不伪造坐标、不影响 edit 成功/失败语义。
- [x] 不改 Pi 源码、Pager renderer 或 ACP 公共协议；adapter 仅消费 Pi 已提供的 result details。
- [x] 覆盖多区域、不同绝对行号的 regression，并通过 adapter、Pager binary check 与 build。

## 关键决策

| 决策 | 理由 |
|------|------|
| 使用 Pi 结果的 `details.patch` | Pi 已是文件变更和坐标的事实来源；无需 adapter 读取用户文件或修改 Pi RPC。 |
| 无法精确匹配时省略 meta | 原行为默认第 1 行虽不精确，但比捏造错误文件坐标安全。 |
| 在 tool end 更新 content | start event 没有 Pi 的 result details；完成 event 才拥有权威 patch。 |
| 不改 Pager renderer | `extract_edit_hunks` 与 renderer 已支持 `Diff.meta.new_line`，缺失数据才触发 fallback。 |

## Notes

最小修复落在 adapter：Pi edit 的 `details.patch` 已从原文件生成，带 `@@ -old,count +new,count @@` 坐标。adapter 用 Pi 已保证唯一的 `oldText` 在对应 hunk 定位，再从相邻新侧行推导 new-file 坐标，因此不会被重复 `newText` 误配；向 ACP Diff 写入 `old_line` / `new_line`。无 patch、无 hunk 或找不到唯一 old text 时不写 metadata。live 与 history ToolCall completion 都使用该投影；start 阶段仍保留原即时 preview。
